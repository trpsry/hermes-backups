# Hermes Wiki Beginner Guide

> คู่มือผู้เริ่มต้นใช้งาน Hermes Wiki
> อธิบายตั้งแต่เริ่มสร้าง จนถึงวิธีใช้จริง สำหรับพี่จอนและสำหรับเฮอ

## สรุปสั้น ๆ
Hermes Wiki คือชั้นความรู้ใหม่ที่ถูกสร้างขึ้นมาใน Vault เพื่อให้เฮอค้นเรื่องสำคัญของระบบ Hermes ได้ง่ายขึ้น จดเพิ่มได้เป็นระเบียบขึ้น และย้อนบริบทได้เองโดยไม่ต้องขุดโน้ตทั้ง Vault มั่ว ๆ ทุกครั้ง

มันไม่ได้มาแทนโน้ตเดิม แต่ทำหน้าที่เป็น **ชั้นสรุป + ชั้นเชื่อมโยง + ชั้นเก็บความรู้ที่โตต่อได้**

---

## 1) ก่อนเริ่มทำ โปรเจกต์นี้มีปัญหาอะไร
ก่อนมี Hermes Wiki ความรู้เกี่ยวกับ Hermes กระจายอยู่ในหลายโน้ต เช่น
- กฎหลักของเฮอ
- วิธีทำงานของ Hermes
- ข้อมูลเครื่อง Android/Termux
- วิธีเลือกโมเดล
- recovery notes
- MOC และโน้ตอ้างอิงอื่น ๆ

ปัญหาคือ:
- รู้ว่ามีข้อมูลอยู่ แต่หาไม่เร็ว
- ต้องจำเองว่าเรื่องไหนอยู่โน้ตไหน
- เวลา context บาง ต้องเปิดหลายโน้ตซ้ำ
- ไม่มีหน้ากลางที่บอกว่า "ตอนนี้มีความรู้อะไรเก็บไว้แล้วบ้าง"
- ไม่มีมาตรฐานว่าถ้าจะให้เฮอสร้างความรู้เพิ่ม ควรเขียนแบบไหน

พูดง่าย ๆ คือ **มีความรู้ แต่ยังไม่มีระบบจัดความรู้สำหรับ AI ใช้งานจริง**

---

## 2) เป้าหมายของ Hermes Wiki คืออะไร
เป้าหมายหลักมี 4 ข้อ

1. ให้เฮอหาเรื่องสำคัญของ Hermes ได้เร็วขึ้น
2. ให้เฮอรู้ว่าควรเก็บความรู้ใหม่ไว้ตรงไหน
3. ให้พี่จอนเปิดอ่านแล้วเข้าใจภาพรวมได้ง่าย
4. ให้ระบบโตต่อได้โดยไม่ทำลายโน้ตเดิม

แนวคิดที่ใช้คือแบบ **Karpathy LLM Wiki**
ซึ่งแปลแบบบ้าน ๆ คือ:
- มีที่เก็บของดิบ
- มีหน้าสรุปความรู้
- มีหน้ากติกา
- มีหน้าสารบัญ
- มีบันทึกว่าแก้อะไรไปบ้าง

---

## 3) ตั้งแต่เริ่มจนพร้อมใช้งาน ทำอะไรไปบ้าง

### ขั้นที่ 1: วางโครงก่อน
เริ่มจากสร้างโครงหลักของ Wiki ขึ้นมาใหม่ใต้โฟลเดอร์
`05 Areas/Hermes/Wiki/`

สิ่งที่สร้างก่อนเลยคือ
- `SCHEMA.md` = กติกาการเขียน wiki
- `index.md` = สารบัญกลาง
- `log.md` = บันทึกการเปลี่ยนแปลง
- `_roadmap.md` = แผนและสถานะ

แล้วแยกหมวดหลักออกเป็น 5 หมวด
- `raw/`
- `entities/`
- `concepts/`
- `comparisons/`
- `queries/`

### ขั้นที่ 2: เอาต้นแบบความคิดเข้ามาก่อน
มีการเพิ่ม raw source สำหรับแนวคิด Karpathy LLM Wiki เพื่อใช้เป็นต้นแบบว่าโครงนี้ควรหน้าตาแบบไหน และควรทำงานอย่างไร

### ขั้นที่ 3: เริ่ม seed หน้าแรกที่จำเป็นจริง
สร้างหน้าความรู้จริงชุดแรกจาก canonical notes เดิม โดยไม่ copy ทั้งหน้า

ชุดแรกที่สร้าง เช่น
- backend ของ Hermes บน Android/Termux
- operating layer ของเฮอ
- model routing
- knowledge entry map

### ขั้นที่ 4: ตรวจลิงก์และเก็บงานให้ใช้ได้จริง
หลังสร้างหน้าแรกแล้ว มีการทำ health check หรือตรวจสุขภาพของ wiki เช่น
- ลิงก์พังไหม
- โครงไฟล์ตรงกับ index ไหม
- tags และ frontmatter สม่ำเสมอไหม
- มีข้อมูลเสี่ยงหลุดไหม

### ขั้นที่ 5: ขยาย content รอบสอง
เพิ่มหน้า concept ที่ช่วยให้ใช้งานจริงได้มากขึ้น เช่น
- Daily Learning Loop
- Skills Health Checklist
- Restore Playbook
- Obsidian Usage Guide

### ขั้นที่ 6: cleanup รอบสุดท้าย
รอบนี้คือเก็บงานละเอียด เช่น
- ปรับ tags ให้ตรง taxonomy
- ปรับ README ของแต่ละหมวด
- เพิ่ม core file policy ใน `SCHEMA.md`
- ปรับ roadmap ให้ไม่ค้าง phase เก่า
- ยืนยันว่าลิงก์ไม่พัง และจำนวนหน้าใน index ตรงกับไฟล์จริง

### ขั้นที่ 7: เขียนคู่มือฉบับนี้
หน้าที่พี่จอนกำลังอ่านอยู่นี้ คือคู่มือสรุปโครงการตั้งแต่เริ่มจนพร้อมใช้งาน เพื่อให้ทั้งคนและเฮอใช้ต่อได้ง่าย

---

## 4) อะไรเปลี่ยนแปลงไปบ้าง

### ก่อน
ก่อนหน้านี้ ความรู้ Hermes ส่วนใหญ่กระจายอยู่ในโน้ตหลายใบทั่ว Vault
ลักษณะคือ
- มีของจริงอยู่แล้ว
- แต่ไม่เป็นระบบสำหรับเรียกใช้เร็ว
- ยังไม่มีชั้นสรุปสำหรับ AI
- ยังไม่มี loop สำหรับเก็บความรู้ใหม่กลับเข้า Vault

### หลัง
ตอนนี้มีชั้นใหม่ที่ชื่อ Hermes Wiki ซึ่งทำหน้าที่เป็นระบบความรู้เฉพาะสำหรับ Hermes

โครงใหม่มีประมาณนี้

```text
05 Areas/Hermes/Wiki/
├── SCHEMA.md
├── index.md
├── log.md
├── _roadmap.md
├── raw/
├── entities/
├── concepts/
├── comparisons/
└── queries/
```

และภายในมีหน้าความรู้จริงที่พร้อมใช้งานแล้ว

### จำนวนและสถานะล่าสุด
- Wiki pages ทั้งหมด: 19 หน้า
- broken wikilinks = 0
- index ตรงกับไฟล์จริง
- โครงหลักพร้อมใช้งานจริงแล้ว

---

## 5) โครงสร้างใหม่ดีขึ้นยังไง

### 1. มีหน้ากลางให้เริ่มเสมอ
`index.md` กลายเป็นจุดเริ่มต้นกลาง
เวลาไม่รู้จะเปิดอะไร ให้เริ่มที่นี่ก่อน

### 2. มีมาตรฐานให้เฮอเขียนต่อได้
`SCHEMA.md` บอกกติกาชัดว่า
- หน้าใหม่ต้องมีอะไร
- ห้ามทำอะไร
- tags แบบไหนถึงใช้ได้
- หน้าแบบไหนควรอยู่หมวดไหน

### 3. แยกหมวดตามหน้าที่จริง
- `raw` = ของดิบ
- `entity` = สิ่ง/ระบบ/องค์ประกอบที่เป็นตัวตนชัด
- `concept` = แนวคิดหรือกฎ
- `comparison` = หน้าเปรียบเทียบ
- `query` = คำตอบที่ควรเก็บระยะยาว

ทำให้สร้างความรู้ใหม่ได้เป็นระเบียบมากขึ้น

### 4. ไม่ทำลายโน้ตเดิม
โปรเจกต์นี้ใช้แนว non-destructive
แปลว่า **ไม่ย้าย ไม่ rename ไม่ทับ canonical notes เดิมมั่ว ๆ**
Wiki เป็นชั้นเสริม ไม่ใช่การรื้อบ้านทั้งหลัง

### 5. ลิงก์และการดูแลสุขภาพดีขึ้น
มีการตรวจ broken wikilinks, count, sources, updated, และความเสี่ยงข้อมูลหลุด
จึงไว้ใจการใช้งานระยะยาวได้มากขึ้น

### 6. รองรับการโตต่อในอนาคต
จากนี้ถ้ามีคำตอบดี ๆ หรือความรู้ใหม่เกิดขึ้น ก็มีทางเก็บกลับเข้า `queries/` หรือ `comparisons/` ได้เลย

### 7. รู้ชัดว่าอะไรไม่ควรเอาเข้ามา
บางเรื่องถูกตั้งใจไม่ seed เช่น
- `team-ai-workflow`
- `hermes-sub-agent`
- `Finance Wallet Guard`

เพราะเสี่ยง stale หรือมีข้อมูลอ่อนไหว
อันนี้คือข้อดีเหมือนกัน เพราะช่วยกันไม่ให้ wiki กลายเป็นที่เก็บของเสี่ยง

## 6) วิธีใช้สำหรับพี่จอน

### เริ่มตรงไหนก่อน
ให้เริ่มที่
`05 Areas/Hermes/Wiki/index.md`

หน้านี้เป็นเหมือนสารบัญกลาง
ถ้าจำไม่ได้ว่าข้อมูลอยู่ตรงไหน เปิดหน้านี้ก่อนเสมอ

### ถ้าอยากรู้เรื่องไหน ให้ไปทางนี้
- อยากรู้ภาพรวมการทำงานของเฮอ → `concepts/hermes-operating-layer.md`
- อยากรู้ว่า Hermes รันอยู่บนอะไร → `entities/hermes-termux-android-backend.md`
- อยากรู้ว่าควรอ่านบริบทจากไหนก่อน → `queries/hermes-knowledge-entry-map.md`
- อยากรู้ว่าทำไมใช้โมเดลนี้กับงานแบบนี้ → `comparisons/hermes-agent-model-routing.md`
- อยากดูประวัติการเปลี่ยนแปลงของ wiki → `log.md`
- อยากดูแนวทางและกติกาของระบบ → `SCHEMA.md`

### พี่จอนควรใช้มันแบบไหน
ใช้ wiki เป็น
- แผนที่
- คู่มือเร็ว
- จุดเริ่มต้นก่อนคุยกับเฮอ
- ที่ย้อนกลับมาดูว่าระบบนี้ถูกจัดความรู้ไว้ยังไง

### พี่จอนไม่จำเป็นต้องทำอะไรเอง
พี่จอนไม่ต้องมานั่งจัด tags หรือดู schema เองก็ได้
เปิดอ่านเฉพาะหน้าที่อยากรู้ได้เลย
ส่วนงานเก็บ/อัปเดต/จัดระบบ หนูจะจัดให้ค่ะ

## 7) วิธีใช้สำหรับเฮอ

### เวลาเริ่มงาน
ลำดับที่ควรทำคือ
1. ใช้ memory + current chat ก่อน
2. ถ้ายังไม่พอ เปิด `index.md`
3. เลือกหน้าที่เกี่ยวข้อง
4. ถ้ามีความรู้ใหม่ที่ควรเก็บ ให้ค่อยเพิ่มกลับเข้า wiki

### เวลาอยากสร้างหน้าใหม่
เฮอควรทำตามลำดับนี้
1. อ่าน `SCHEMA.md`
2. ถามก่อนว่าหน้านี้ควรเป็น raw / entity / concept / comparison / query
3. สรุปใหม่ ไม่ copy note เดิมทั้งหน้า
4. ใส่ `sources:` กลับไปหาโน้ตต้นทาง
5. เพิ่มหน้าใน `index.md`
6. ลงเหตุการณ์ใน `log.md`
7. เช็กว่าลิงก์ไม่พัง

### เวลาอยากตอบคำถามจากความรู้เดิม
- เริ่มจาก `index.md`
- ถ้ามีหน้าอยู่แล้ว ให้ใช้หน้านั้นก่อน
- ถ้าคำตอบครั้งนี้ดีและมีค่าระยะยาว ให้เก็บกลับเข้า `queries/` หรือ `comparisons/`

### เวลาเจอข้อมูลเสี่ยง
ถ้าเนื้อหามี
- token
- route ละเอียด
- credential
- live endpoint
- ข้อมูลการเงินส่วนตัว

ห้ามเอาเข้า wiki ตรง ๆ
ให้ link ไป canonical note หรือสรุปแบบปลอดภัยแทน

## 8) คำศัพท์สำคัญแบบง่าย ๆ
- **Schema** = กติกาการเขียนและดูแล wiki
- **Index** = สารบัญกลาง
- **Log** = สมุดบันทึกว่าเปลี่ยนอะไรไปบ้าง
- **Raw** = ของดิบ/ต้นทาง
- **Seed** = หน้าแรก ๆ ที่คัดมาสร้างก่อน
- **Lint / Health check** = ตรวจสุขภาพว่าโครงยังเรียบร้อยไหม
- **Canonical note** = โน้ตตัวจริงที่เป็นแหล่งอ้างอิงหลัก
- **Query-back loop** = เวลาตอบคำถามดี ๆ แล้วเก็บกลับเข้า wiki

---

## 9) ข้อควรระวัง
1. Wiki ไม่ใช่ source of truth ทุกเรื่อง
   - ถ้าขัดกับ canonical note ให้ยึด canonical note เป็นหลัก
2. ห้าม copy note เดิมทั้งหน้าเข้ามา
3. ห้ามยัดข้อมูลลับหรือข้อมูลเสี่ยงเข้า wiki
4. ห้ามสร้างหน้าใหม่พร่ำเพรื่อถ้ายังไม่จำเป็น
5. ถ้าจะขยายต่อ ให้ขยายจากการใช้งานจริง ไม่ใช่สร้างเผื่อเยอะเกิน

---

## 10) สรุปสุดท้าย
ตอนนี้ Hermes Wiki เดินมาถึงจุดที่เรียกว่า **พร้อมใช้งานจริง** แล้ว

ความหมายของคำว่าพร้อมใช้งานในที่นี้คือ
- มีโครงหลักครบ
- มีหน้าความรู้หลักครบพอเริ่มใช้
- มีกติกาชัด
- มีสารบัญกลาง
- มีบันทึกการเปลี่ยนแปลง
- มีการตรวจสุขภาพแล้ว
- ไม่มีลิงก์พัง
- รู้ชัดว่าอะไรควรเก็บ และอะไรไม่ควรเก็บ

จากนี้งานจะไม่ใช่การซ่อมโครงหลักแล้ว
แต่เป็นการ **ค่อย ๆ ใช้งานจริง แล้วเติมความรู้ใหม่อย่างมีวินัย**
